# 概述

    目前市面上有很多这类的AI聊天， 例如: 小爱同学，小度，siri等。但是这些机器人都是很泛的聊天，就是问的东西没有强的上下文，一旦问的东西有很多领域都有涉及的， 回答的时候就不能准确的回答了，给我们的感觉，说的什么鬼东西，本质上还是，提问者的上下文和回答者的上线文是没有统一的。 一旦限制了上下文， 就意味着进入了垂直领域了，例如医疗助手（会思考的呦），教学机器人，等等。

# 技术实现核心流程

    整个流程可以分为三个主要阶段：输入处理（STT/ASR）、核心处理（AI）、输出处理（TTS）

## 输入处理（设备 → ASR）

    1. 语音采集	设备（如手机、IoT 麦克风）捕获语音。原始模拟信号

    2. 编码压缩	设备上的编码器（Encoder）将语音信号编码。	
        AAC (Advanced Audio Coding)	
        目的： 节省带宽和存储空间。AAC 是目前主流的高效有损音频压缩格式。

        有必要的话封装成容器格式(MP4,FLV)都行(可选，跟使用那家ASR/STT的接口有关)

    3. 数据传输	设备将 AAC 数据包通过网络（TCP/UDP）发送给服务器。	AAC 数据流.

    4. 服务器解码	服务器端的解码器（Decoder）接收 AAC 数据流，并将其解压缩(解码)。解码生成PCM (Pulse Code Modulation)	目的： ASR 模型只能识别原始的、未压缩的 PCM 波形数据。所有压缩格式（AAC/MP3/Opus 等）都必须解码为 PCM。(通意千问是这样的， Google，aws 是要求传容器格式的，哈哈哈)

    5. ASR 处理	服务器端的 ASR 模型开始识别 PCM 格式的语音数据。	文本	ASR 模型输入 PCM → 输出文本。

## 阶段二：核心处理（AI 核心 → TTS）

    6. 文本处理，服务器处理 ASR 识别出的文本（例如：执行命令、查询数据库、生成回复）。核心逻辑是将识别出来的文本输入到具有垂直领域的上下文提示词的agent里面去，等待获取LLM 返回的结果，

    7. TTS 合成，	服务器端的 TTS 模型（语音合成器）将生成的文本转换为语音。		TTS 模型也必须以 PCM(MP4容器) 格式输出。 PCM 是数字音频的基准格式，包含了波形的所有细节，可直接转换为声音。

## 阶段三：输出处理（TTS → 设备）

    8. 服务器编码	服务器端的编码器（Encoder）将 TTS 产生的 PCM 语音进行压缩。	AAC	目的： 再次压缩数据流，减少网络传输带宽，提高响应速度。

    9. 数据传输	服务器将 AAC 数据包通过网络发送给设备。	AAC 数据流.

    10. 设备解码，设备端的解码器（Decoder）接收 AAC 数据流并进行解压缩。	PCM	目的： 播放器硬件（DAC, 数模转换器）只能处理原始的 PCM 信号。

    11. 语音播放	设备将 PCM 信号通过 DAC 转换成模拟信号，驱动扬声器播放。原始模拟信号
