# 简介

  在向量数据库中，索引（Indexing） 的核心目标是解决“维度灾难”。在处理成千上万维的向量时，传统的 B-tree 等索引失效，必须采用 近似最近邻搜索（ANN, Approximate Nearest Neighbor） 算法。

# 向量数据库索引的实现主要分为四大流派

## 基于图的索引：HNSW (Hierarchical Navigable Small World)

  这是目前最流行、性能最均衡的算法，也是 Milvus、Pinecone 和 ElasticSearch 的默认选择。

  实现原理：它构建了一个多层的跳表结构。最底层包含所有向量（节点），层数越高，节点越稀疏。

  搜索过程：搜索从最高层开始，利用“小世界”特性快速定位到目标区域，然后逐层向下切换，直到在最底层找到最精确的邻居。

  优点：查询速度极快（对数级复杂度），召回率（Accuracy）极高。

  缺点：内存消耗巨大，因为需要存储大量的图连接边。

## 基于聚类的索引

IVF (Inverted File Index)IVF 是为了解决内存问题而设计的“倒排索引”变体。

实现原理：聚类：利用 K-Means 算法将向量空间划分为 $K$ 个簇（Clusters）。

倒排列表：记录每个簇中心点以及属于该簇的所有向量 ID。

搜索过程：先找到与查询向量最近的几个簇中心，然后只在这些簇内进行遍历。

优点：大幅减少了计算量，显著降低磁盘/内存占用。

缺点：如果搜索的簇不够多，可能会漏掉真正的最近邻（召回率略低）。

## 基于压缩/量化的索引：PQ (Product Quantization)

  这是一种极其硬核的“空间换时间”技术，通常与 IVF 配合使用（即 IVF_PQ）。

  实现原理：将高维向量切分成多个子向量，并对每个子向量进行聚类，用一个极短的“码本（Codebook）”索引来代替原始向量。

  优点：极致的压缩比。可以将几百 GB 的数据压缩到几十 GB，甚至可以直接在内存中处理海量数据。

  缺点：计算距离时存在精度损失。

## 基于树的索引：Annoy (Approximate Nearest Neighbors Oh Yeah)

  由 Spotify 开发，广泛应用于推荐系统。

  实现原理：通过随机超平面不断将空间一分为二，最终形成多棵二叉树。

  优点：索引文件可以映射到内存（Static Index），支持多进程共享，非常适合只读场景。

  缺点：对于极高维度的向量，性能下降明显。

# 核心算法对比表

|索引类型|代表算法|核心优势|缺点|适用场景|
|--|--|--|--|--|
|图索引|HNSW|速度最快、精度最高|内存开销极大|对性能要求极高的实时搜索|
|聚类索引|IVF|内存占用低，结构简单|召回率受聚类质量影响|中等规模、内存有限的场景|
|量化索引|PQ|极高压缩比|精度有损|海量数据（十亿级）存储|
|树索引|Annoy|内存占用极小，静态文件|无法实时增量更新|离线推荐、移动端搜索|


# 更前沿或更垂直的索引实现

## 基于哈希的索引：LSH (Locality Sensitive Hashing)

  这是早期的主流算法，虽然在现代高性能数据库中用得少了，但在某些特定的大规模近似搜索中依然有价值。

  实现原理：利用特殊的哈希函数，将空间中距离较近的点以较高的概率映射到同一个“桶（Bucket）”中。

  特点：

    优点：计算极其简单，非常适合大规模数据的初步过滤。

    缺点：为了保证精度，通常需要维护大量的哈希表，导致内存开销激增，且召回率往往不如 HNSW。

## 针对磁盘优化的索引：DiskANN (及衍生算法如 SPANN)

  传统的 HNSW 必须把所有数据塞进内存，这在处理十亿（Billion）级别数据时极其昂贵。DiskANN 是为了把向量“存入 SSD 还能跑得飞快”而设计的。

  实现原理：

  Vamana 图：一种比 HNSW 具有更小直径的图结构。

  压缩存储：在内存中只存一份压缩后的向量（用于初步计算），而将完整的原始向量存储在 SSD 上。

  单次 I/O 读取：优化数据布局，确保搜索时只需极少数次（通常是 1 次）磁盘读取即可获取候选集。

  优势：成本极低。你可以在一台只有 64GB 内存的服务器上搜索 10 亿条向量。

## 基于学习的索引：ScaNN (Scalable Nearest Neighbors)

  这是 Google 开发的算法，在多项基准测试中表现极其出色（尤其是在 CPU 上）。
  
  实现原理：它引入了一种各向异性量化（Anisotropic Quantization）。

  传统的量化只关注减少整体误差，而 ScaNN 的量化更关注如何保持向量之间的“内积排名”。

  优势：在最大内积搜索（MIPS）任务中，它能提供比 PQ 更高的精度和更快的速度。

## 基于硬件原生加速的索引：RAFT (GPU-based)

  由 NVIDIA 推出的库，将传统的向量索引逻辑直接重写为 GPU 核函数。

  实现原理：利用 GPU 的大规模并行能力，重新实现了 IVF-PQ 和 Brute-force 搜索。

  黑科技：它能实现“暴力搜索（Flat）”甚至比 CPU 上的“近似搜索”还要快。在 2026 年，很多实时性要求极高的金融或防御系统会选择直接用 GPU 跑全量搜索。

## 复合型索引 (Hybrid Indexing)

  目前的趋势是不再使用单一算法，而是采用“组合拳”：

  IVF_HNSW：先用聚类（IVF）把空间分成 1000 块，然后在每一块内部建立 HNSW 图。这样既保留了图的高速，又解决了全量 HNSW 无法横向扩展的问题。

  NSG (Navigating Spreading Graph)：在 HNSW 基础上优化了图的连通性，去掉了冗余的边，进一步压缩了内存。

## 如何选择这些“其它”索引？

|如果你的需求是...|推荐索引|理由|
|--|--|--|
|省钱，数据量级超过 1 亿|DiskANN / SPANN|充分利用廉价的 SSD，而不是昂贵的内存。|
|极致性能（不管显卡多贵）|RAFT (GPU Index)|吞吐量是 CPU 的几十倍。|
|处理最大内积搜索（推荐系统）|ScaNN|各向异性量化更适合排名任务。|
|极简实现或静态数据|LSH|算法简单|易于在非专用数据库中实现。|

## 如何监控索引的性能？

  在实战中，你不仅要选索引，还要关注这三个指标的平衡：

  Recall (召回率)：AI 找得准不准？

  QPS (每秒查询数)：系统撑不撑得住？

  Latency (延迟)：单个请求快不快？
