# 概述

    高度自动化和元数据驱动的软件开发流程或架构模式，特别适用于构建低代码/无代码平台、快速应用开发（RAD）工具或领域特定语言（DSL）驱动的系统。
    这个流程可以概括为：所见即所得的设计 (可视化编辑器) -> 抽象模型解析 (元数据引擎) -> 自动化代码生产 (代码生成器) -> 部署与执行 (运行时容器)。
    让我们来详细分解这个流程的每个组成部分：

## 简介
    
    我们经常在项目中使用流程式，依赖式等DAG形式的可视化编辑。常见的有: node-red,aws framemake, dolphins schedule，serverless framework 等，就会出现画DAG图来串因爱逻辑。
    画DAG这类的依赖关系一般是用HTML5 Canvas 或 SVG.

## 连线的公共特性与实现细节

### 锚点计算: 

    连线并非直接从一个节点的中心点连接到另一个节点的中心点。它们通常是从节点的输出端口（右侧小方块）连接到另一个节点的输入端口（左侧小方块）。这意味着在绘制连线时，需要精确计算这些端口的屏幕坐标作为连线的起点和终点。

### 路径规划: 
    
    简单的连线可能是直线，但为了美观和避免遮挡，连线通常是折线或贝塞尔曲线。它们会尝试避开其他节点，使得流程图更清晰。这涉及到复杂的路径规划算法。

### 层级管理 (Z-order)： 

    连线和形状都有层级关系，例如一条连线可以从某个形状上方经过，也可以从下方经过。通过管理 SVG 元素的 z-index 或它们在 SVG DOM 中的顺序来控制这些层级。

### 交互性:

#### 选择连线: 

    用户可以点击连线来选择它，通常连线会高亮显示。

#### 删除连线: 
    
    选择后按 Delete 键可以删除连线。

#### 拖拽修改连线: 
    
    有些高级编辑器允许你拖拽连线上的某个点来改变其形状，或者将其重新连接到另一个端口。

#### 性能优化: 

    对于大型 Flow，可能有成百上千条连线。高效的渲染（只重绘发生变化的连线）和交互响应是关键。

## 元数据管理

XML/JSON 格式： 图表数据通常以 XML 或 JSON 格式存储。这个数据模型详细描述了：

每个形状的类型、ID、位置、大小、样式属性、文本内容。

每条连线的源节点 ID、目标节点 ID、连接点信息（是浮动还是固定，具体哪个点）、路径点坐标、样式属性、标签内容。

部署与保存： 当用户保存图表时，这个结构化的数据模型会被保存到本地文件、Google Drive、OneDrive 等云存储服务中。当重新打开图表时，解析这个 XML/JSON 数据，并在前端重新渲染图表。
